모델링
	추상적인 주제를 DB 시스템에 맞게 설계하는 것

	예) 영화관
	
	1. 요구 분석
		무언가 개발하기 위해 필요한 데이터들, 처리해야 할 데이터들이 어떤 것들이 있는지
		
		상영관에 대한 정보(위치, 이름, 좌석수, ...), 영화에 대한 정보(감독, 배우, 런닝타임, ...)
		위치, 이름, 좌석수, 감독, 배우, 런닝타임, ... 

	2. 개념적 설계(개념 모델링)
		상영관에 대한 개념 정의: 상영관이라는 위치와, 수용 가능한 좌석 수 등등...
		영화라는건 제작한

	3. 논리적 설계(논리 모델링)
		개념적 설계로 나온 결과를 기반으로 데이터베이스에 저장할 수 있는 형태로 구조화시키는 단계(추상적으로 보이는 테이블 형태를 구조화시킴)

		상영관		시간표		영화	
		
		상영관코드(PK)	영화코드(FK)	영화코드(PK)
		위치		상영관코드(FK)	감독
		이름		시작 시간		배우
		좌석수		끝나는 시간		런닝타임

	4. 물리적 설계(물리 모델링)
		상영관코드	: varchar(300) primary key
		위치	: varchar(1000) not null
		이름	: varchar(300) not null
		좌석수	: int
		...

	5. 구현

정규화(모델링을 하는 동시에)
	삽입, 수정, 삭제의 이상현상을 제거하기 위함
	데이터의 무결성을 위해서 진행
	데이터의 중복이나 잘못된 데이터를 최소화 하기 위함
	이러한 이상 현상을 해결하기 위해 정규화를 한다. -> 테이블을 나누어가는 과정
	-> 대부분 3차 정규화까지 진행

	아이디	이름	전공	ㅣ	수업코드	수업명		최대인원
	apple	김사과	컴공과	ㅣ	C001	너무쉬운자바	30
	apple	김사과	전자과	ㅣ	C002	너무쉬운DBMS	35 <- 다른 데이터도 수정을 해야할 때, 수정 이상 현상
	banana	반하나	수학과	ㅣ	C002	너무쉬운DBMS	35 <- 수업명을 삭제하는데 나머지 데이터까지 삭제될 때, 삭제 이상 현상
	xxx	xxx	xxx	ㅣ	C003	너무쉬운HTML	30 <- 의미없는 데이터들을 넣어야할 때, 삽입 이상 현상


1차 정규화
	하나의 컬럼에는 값이 한 개씩만 있어야 한다.
	
	[apple	김사과	바지1, 바지2, 코드] 를 아래와 같이 고친다.

	apple	김사과	바지1
	apple	김사과	바지2
	apple	김사과	코트

2차 정규화
	각 테이블의 모든 컬럼들이 서로 관계가 있어야 한다. 

	이름과 상품명이 관계가 있는지 확인 (엔티티의 관점에서 하나의 대상(개체)으로 묶일 수 있는지)
	-> 이름과 상품명의 테이블을 나눈다.
	apple	김사과	ㅣ	바지1
			ㅣ	바지2
			ㅣ	코트
	

3차 정규화
	하나의 컬럼은 그 외 다른 컬럼을 결정하면 안된다. pk가 결정하는건 괜찮다.

반정규화
	활용하는 쪽에서 데이터를 관리, 사용하기에 더 편하다면
	실용적인 이유에서 반대로 정규화를 진행하지 않는 방법

	만약 배우들의 나이 정보가 추가된다면, 정규화를 해야함.

DML
	Data Manipulation Language - 데이터 조작어
	실질적으로 데이터들을 CRUD 하는 언어

	데이터 추가
		insert into 테이블명 [(컬럼명1, 컬럼명2, ...)]
		values(값1, 값2, ...);

	데이터 삭제
		delete from 테이블명 [where 조건식]; <- [where 조건식]이 없으면 모두 삭제가 되니 조심!!


	데이터 수정
		update 테이블명
		set 기존컬럼명 = 바꿀값, 기존컬렴명 = 바꿀값, ...
		[where 조건식];			<- [where 조건식]이 없으면 모두 수정이 되니 조심!!


	데이터 조회(종류가 많음)
		select 컬럼명1, 컬럼명2, ... from 테이블명 [where 조건식];


where 조건절
	where 조건식
	행(데이터)ㅔㅇ 대한 조건식을 써주는 조건절
	행을 찾기 위한 조건절
	
	= 	: 같다
	!=, <>	: 다르다
	and	: 두 조건식이 모두 참이면 참
	or	: 두 조건식 중 하나라도 참이면 참
